# استقرار DigiWP Ai Support (Coolify)

یک استک، چند سرویس — همه در [`docker-compose.yaml`](docker-compose.yaml):

| سرویس | چیست | دسترسی |
|---|---|---|
| **hub** | پنل / سایت اصلی (React روی nginx) | عمومی — نصب اصلی شما |
| **server** | **سرور واسط / API** (Node) | عمومی — دامنهٔ اختصاصی خودش (`/health` و `/v1`)؛ هاب هم از `/api` به آن پراکسی می‌کند |
| **db** | **PostgreSQL** — دیتابیس هاب | داخلی |
| **demo** (+ **demo-db**) | یک وردپرس واقعی با افزونهٔ کانکتور نصب‌شده | عمومی — برای دیدن حلقهٔ کامل |

```
مرورگر → hub (nginx)
              ├── فایل‌های استاتیک پنل
              └── /api/* → server:8787/v1/*  → (رله‌ی امضاشده) → کانکتور روی سایت وردپرسی
```

هاب و سرور **یک مبدأ (origin)** هستند، پس CORS و آدرس‌دهی دردسر ندارد.

---

## استقرار روی Coolify

1. **New Resource → Docker Compose** و این ریپو را انتخاب کنید (فایل `docker-compose.yaml` در ریشه).
2. در بخش **Environment Variables** این‌ها را ست کنید:
   - `AUTH_SECRET` — یک رشتهٔ تصادفی بلند (کلید امضای نشست‌ها). **حتماً عوض کنید.**
   - `POSTGRES_PASSWORD` — رمز دیتابیس PostgreSQL هاب.
   - `DEMO_DB_PASSWORD`, `DEMO_DB_ROOT_PASSWORD` — رمزهای دیتابیس دمو.
3. **Deploy**. Coolify برای `hub` و `demo` به‌صورت خودکار دامنه می‌سازد
   (متغیرهای `SERVICE_FQDN_HUB_80` و `SERVICE_FQDN_DEMO_80`). سرور `PUBLIC_BASE_URL`
   را از دامنهٔ هاب می‌گیرد (`${SERVICE_URL_HUB}/api`).
4. دیتابیس PostgreSQL هاب روی والیوم `pg-data` می‌ماند؛ با ری‌دیپلوی پاک نمی‌شود.
   سرور تا آمادهٔ‌کار شدن دیتابیس صبر می‌کند (retry داخلی + healthcheck).

> اگر می‌خواهید سرور دامنهٔ مستقل هم داشته باشد، کافی است `SERVICE_FQDN_SERVER_8787`
> را به env سرویس `server` اضافه کنید — ولی برای کار کردن لازم نیست.

## اجرای محلی (بدون Coolify)

برای اجرای محلی، فایل پورت‌های `deploy/compose.local.yml` را هم صریحاً بدهید تا
سرویس‌ها روی `localhost` باز شوند (این فایل عمداً `docker-compose.override.yaml`
نیست تا Coolify آن را برندارد):

```bash
AUTH_SECRET=dev docker compose -f docker-compose.yaml -f deploy/compose.local.yml up --build
```

- پنل: <http://localhost:8080>
- API / سرور واسط: <http://localhost:8787>  (`/health` و `/v1`)
- وردپرس دمو: <http://localhost:8081>

(پورت‌ها با `HUB_PORT` / `SERVER_PORT` / `DEMO_PORT` قابل تغییرند.)

> Coolify فقط از `docker-compose.yaml` استفاده می‌کند و این فایل را نمی‌بیند —
> پس تداخل پورت روی سرور Coolify پیش نمی‌آید.

## دموی «چطور کار می‌کند»

### ۱) پنل — بلافاصله آماده
سرور در اولین اجرا یک کاربر نمونه می‌سازد:

- **ایمیل:** `maryam@example.com`  ·  **رمز:** `demo1234`

با آن وارد شوید تا کل پنل با ۳ سایت نمونه، هشدارها، آپدیت‌های ریسک‌سنجی‌شده،
بکاپ‌ها و دستیار را ببینید. ثبت‌نام کاربر جدید هم واقعی است و در دیتابیس می‌ماند.

### ۲) اتصال سایت وردپرسِ دمو (حلقهٔ کامل)
1. در پنل: **افزودن سایت** → دامنهٔ دمو را وارد کنید → **ساخت کلید اتصال**.
   یک **رمز مشترک** و **آدرس سرور** به شما می‌دهد (فقط همان یک‌بار).
2. وردپرس دمو را باز کنید (`/wp-admin`)، نصب اولیه را کامل کنید،
   افزونهٔ **WP Claude Bridge** را فعال کنید.
3. **ابزارها → Claude Bridge → Hub Connector Mode**: حالت کانکتور را روشن کنید و
   همان **آدرس سرور** و **رمز مشترک** را وارد و ذخیره کنید.
4. برگردید به پنل و **بررسی اتصال** را بزنید — سرور یک درخواست امضاشده به کانکتور
   می‌زند و اتصال زنده تأیید می‌شود. از این پس افزونه فقط فرمان‌های امضاشدهٔ سرور
   شما را می‌پذیرد و مستقیم روی سایت کاری نمی‌کند.

> افزونه به‌صورت فقط‌خواندنی داخل سرویس دمو مانت شده
> (`./wp-claude-bridge.php` و `./skills`)، پس همان کد این ریپو اجرا می‌شود.

## امنیت تولید
- `AUTH_SECRET` را حتماً یک مقدار تصادفی و مخفی بگذارید.
- برای بار واقعی، `store` مبتنی بر SQLite را با یک دیتابیس مدیریت‌شده جایگزین کنید
  (لایهٔ `server/src/store.js` تنها نقطهٔ تماس است).
- `CORS_ORIGIN` را از `*` به دامنهٔ هاب محدود کنید اگر سرور دامنهٔ جدا دارد.
